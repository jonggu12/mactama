# MacTama — PRD v3.1

> MacBook 센서 기반 메뉴바 펫 앱  
> 버전 3.1 | 작성일 2026-03-31  
> 변경 이력: v3.0 → v3.1 — 제품 정체성 유지, 출시 범위 재정의, 진화 시스템 출시 시점 조정, 메뉴바 UI 단순화, 성향 명칭 톤 보정

---

## 0. 문서 역할

이 문서는 두 가지를 함께 정의한다.

- **제품 PRD**: MacTama가 어떤 앱인지, 왜 다른지, 어디까지 확장될지를 정의
- **출시 PRD**: 첫 버전에서 실제로 무엇을 넣고 무엇을 뒤로 미룰지를 정의

핵심 원칙은 그대로다.

> **그냥 맥북을 쓰다 보면 펫이 내 습관대로 자라 있다.**

다만 첫 출시에서는 이 경험을 가장 작은 범위로 먼저 검증한다.

## 1. 제품 정체성

### 1.1 한 줄 정의

> **내 맥북의 충전, 수면, 과로를 먹고 자라는 메뉴바 펫.**

### 1.2 핵심 차별점

> **맥북 상태에 따라 진짜 살아있는 것처럼 변하고, 내 사용 습관이 펫 성향으로 굳어지는 메뉴바 펫.**

사용자가 맥북을 쓰는 방식이 곧 펫의 삶이다.

- 충전선을 꽂으면 펫이 밥을 먹는다
- 뚜껑을 닫으면 펫이 잠든다
- 밤새 코딩하면 펫에게 다크서클이 생긴다
- CPU를 혹사하면 펫이 달아오른다

사용자가 펫을 따로 관리하는 것이 아니라, 맥북을 쓰는 방식이 캐릭터를 만든다.

### 1.3 무엇이 아닌가

- 수치를 관리해야 하는 육성 게임이 아니다
- 잘 키우면 좋은 펫, 못 키우면 나쁜 펫이 되는 구조가 아니다
- 방치하면 죽는 스트레스 앱이 아니다

**"대충 켜두면 살아있고, 이벤트 때만 반응이 크고, 쓰다 보면 나를 닮아 있다."**

### 1.4 바이럴 포인트

사용자가 자연스럽게 공유하게 되는 문장들:

- "내 맥북 펫은 야근형으로 굳어졌어"
- "너는 왜 집냥이형이냐 나는 완전 야행성임"
- "밤새 쓴다고 내 펫 다크서클 생겼음"

## 4. 성향 시스템 — 제품 레벨 방향

이 장은 **제품이 최종적으로 도달하려는 방향**을 정의한다.  
실제 포함 시점은 10장 릴리스 범위를 따른다.

### 4.1 설계 원칙

진화는 **보상**이다. 강제 관리의 결과가 아니다.  
진화는 **맥북 사용 패턴의 성격화**다. 잘 키웠는지의 판정이 아니다.

유저는 조건을 계산하지 않는다. 그냥 맥북을 쓰다 보면 어느 날 펫이 달라져 있다.  
목표는 "어? 얘 왜 이렇게 됐지?"라는 발견이다.

모든 성향은 부정적이지 않아야 한다.

- Cozy형은 포근하고 안정적이어야 한다
- Overwork형은 힘들지만 멋있고 몰입한 느낌이어야 한다
- Mischief형은 문제아가 아니라 장난기 많고 들쑥날쑥한 느낌이어야 한다

### 4.2 성향 축

| 성향 | 의미 | 톤 |
|------|------|----|
| Cozy | 안정적이고 잘 쉬는 사용자 | 포근함, 둥글음, 순함 |
| Overwork | 밤과 몰입에 익숙한 사용자 | 야행성, 집중, 열기 |
| Mischief | 충동적이고 들쑥날쑥한 사용자 | 장난기, 반항기, 예측불가 |

### 4.3 네이밍 체계

MacTama는 성향을 두 층위로 다룬다.

- **내부 축명**: 코드, 상태머신, 분석 이벤트에서 쓰는 시스템 이름
- **사용자 노출명**: 팝오버, 공유 문구, 알림에서 보이는 감정적 이름

| 내부 키 | 내부 축명 | 사용자 노출명 예시 |
|--------|-----------|--------------------|
| `cozy` | Cozy | 집냥이형 / 수면왕형 |
| `overwork` | Overwork | 야근냥이형 / 코딩머신형 |
| `mischief` | Mischief | 장난냥이형 / 번개냥이형 |

원칙:

- 내부 로직과 계측 이벤트는 영어 키를 사용한다
- 사용자에게는 더 귀엽고 애정 어린 한국어 명칭을 노출한다
- 벌점처럼 느껴질 수 있는 표현은 피한다
- 성향 축명과 최종 진화명은 동일할 필요가 없다

### 4.4 진화 단계

```text
Stage 0 (초기)        Stage 1 (성향 감지)      Stage 2 (성향 고착)
─────────────        ─────────────────       ──────────────────────
아기 맥키            Cozy / Overwork /       성향별 외형 + 반응 차별화
(기본형)             Mischief 힌트 발생
```

### 4.5 Stage 0 → Stage 1 판별 방향

최근 3일간 맥북 사용 패턴을 바탕으로 성향을 추론한다.  
이 숫자는 내부 로직이며, 유저에게 직접 노출하지 않는다.

**Cozy형**

- 하루 평균 수면(뚜껑 닫힘) 시간이 충분함
- 배터리가 자주 바닥나지 않음
- 사용 시간대가 주로 낮에 몰림

외형 힌트:

- 귀가 둥글고 넓어짐
- 눈이 크고 선해짐
- 볼이나 화면 톤이 부드러워짐

**Overwork형**

- 자정 이후 사용 빈도가 높음
- CPU 고부하 이벤트가 반복됨
- 수면 패턴이 짧거나 불규칙함

외형 힌트:

- 귀가 살짝 처짐
- 눈이 반쯤 감기거나 다크서클이 생김
- 케이블이나 열기 이펙트가 붙음

**Mischief형**

- 배터리를 자주 낮게까지 씀
- 사용 패턴이 들쑥날쑥함
- 상호작용이 거칠거나 반응성이 큼

외형 힌트:

- 귀가 뾰족해짐
- 눈빛이 날카롭거나 장난기 있게 바뀜
- 머리 위 번개 또는 작은 장난 아이콘이 생김

### 4.6 Stage 1 → Stage 2 고착

Stage 1 이후 같은 패턴이 일정 기간 지속되면 Stage 2로 고착된다.  
Stage 2에서는 같은 이벤트에도 반응 방식이 달라진다.

| 이벤트 | Cozy형 반응 | Overwork형 반응 | Mischief형 반응 |
|--------|------------|----------------|-----------------|
| 전원 연결 | 행복하게 밥 먹기 | "버텨야지" 표정 | 허겁지겁 달려들기 |
| 뚜껑 닫힘 | 포근하게 잠들기 | 쓰러지듯 잠들기 | 버티다 툭 잠들기 |
| 강한 입력/슬랩 | 깜짝 놀라기 | 무덤덤함 | 반격 자세 |
| CPU 과부하 | 당황해서 식은땀 | 눈빛이 살아남 | 연기 뿜으며 과열 반응 |

### 4.7 메뉴바에서 보여야 하는 변화

메뉴바는 14×14px이므로, 변화는 크게 그리지 말고 **실루엣 중심**으로 설계한다.

- 귀 실루엣: 둥글 / 처짐 / 뾰족
- 눈 모양: 선함 / 반쯤 감김 / 날카로움
- 머리 위 아이콘: 없음 / 별 / 번개
- 색 1~2개 변화: 볼 색, 화면 색, 열기 색

### 4.8 진화 연출

진화 순간은 강한 감정 포인트다.

1. 메뉴바 아이콘이 3회 번쩍임
2. macOS Notification 발송: "맥키가 달라졌어요"
3. 팝오버 자동 오픈 후 3초 내외 변신 연출
4. 새 성향 이름과 짧은 설명 표시

예시:

> 야근냥이 — 밤의 작업 리듬에 익숙한 타입으로 자랐어요

## 5. 내부 상태 리듬

### 5.1 원칙

> MacTama는 숫자를 관리하는 앱이 아니라, 상태가 자연스럽게 바뀌는 펫 앱이다.

- 사용자가 숫자를 관리하게 만들지 않는다
- 팝오버에 숫자 HUD를 전면에 노출하지 않는다
- 죽음이나 실패를 중심 경험으로 두지 않는다
- 하루 종일 켜놔도 과한 스트레스를 주지 않는다
- 변화는 이벤트와 시간 흐름에 따라 자연스럽고 이해 가능해야 한다

### 5.2 v1 기본 상태 축

v1에서는 숫자보다 **내부 단계형 상태**를 우선 사용한다.  
필요하다면 나중에 숫자형 내부 값으로 확장하되, 초기에는 아래 축만 있으면 충분하다.

| 상태 축 | 의미 | v1 권장 단계 |
|--------|------|--------------|
| 에너지 | 충전 / 배터리 상태에 대한 펫 해석 | 안정 / 보통 / 부족 |
| 기분 | 현재 전반적인 컨디션 | 좋음 / 보통 / 무덤덤 |
| 피로 | 시간 경과와 수면의 영향 | 말짱 / 나른함 / 피곤함 |

### 5.3 사용자에게 보이는 방식

이 내부 상태는 숫자로 직접 노출하지 않고, **짧은 문장과 표정 변화**로 보여준다.

예:

- "전원 연결됨"
- "배터리로 버티는 중"
- "잠자는 중"
- "방금 깨어남"
- "조금 지친 표정이에요"

### 5.4 감정 우선순위

```text
1. critical  — 에너지 부족 또는 피로 누적이 큰 상태
2. sleeping  — 뚜껑 닫힘
3. lowEnergy — 배터리 기반 에너지 저하 상태
4. tired     — 피로 누적 상태
5. bright    — 기분이 좋은 상태
6. normal    — 기본값
```

### 5.5 성향 고착 후 보정

| 성향 | 보정 |
|------|------|
| Cozy | 수면 후 회복감이 더 빠르게 느껴짐 |
| Overwork | 과부하 상황에서도 쉽게 지치지 않음 |
| Mischief | 거친 상호작용에 덜 예민하게 반응함 |

## 6. 센서 → 이벤트 매핑

### 6.1 v1에 포함하는 안정적인 센서

| 트리거 | API | 펫 반응 |
|--------|-----|---------|

## 8. UI 명세

### 8.1 메뉴바

기본 상태에서는 **캐릭터만 노출**한다.  
항상 숫자를 띄우지 않는다. 숫자는 게임 HUD처럼 보일 위험이 있다.

```text
기본:      [캐릭터 14px]
이벤트:    [⚡ 캐릭터] 충전중
기상:      [☀ 캐릭터] 깨어남
진화:      [✨ 캐릭터] 달라졌어요
```

원칙:

- 평상시 메뉴바는 최대한 조용해야 한다
- 이벤트 때만 2~3초 짧게 텍스트를 띄운다
- 성향과 상태 해석은 팝오버에서 본다

### 8.2 팝오버 (200×300px)

- 캐릭터 대형 애니메이션 64×64px
- 현재 상태 한 줄
- 짧은 상태 설명 한 줄
- 최근 이벤트 로그 3건
- 현재 성향 자리 또는 "분석 중" 상태
- 공유 버튼은 v1.1 이상에서 추가

원칙:

- 팝오버는 대시보드가 아니라 캐릭터 상태 카드여야 한다
- 숫자 게이지보다 짧은 문장과 최근 이벤트가 우선이다
- 시스템 값 자체보다 "펫이 지금 어떤 상태인가"를 해석해서 보여준다

공유 문구 예시:

```text
나의 맥북 펫은 [야근냥이] 느낌으로 자라고 있어요.
맥북 사용 습관이 드러나는 메뉴바 펫, MacTama
```

## 9. 계측 및 분석

### 9.1 측정 원칙

MacTama의 초기 검증 목표는 "살아있는 느낌"과 "계속 켜두는 습관"이다.  
따라서 단순 설치 수보다도 첫 반응 경험, 재방문, 팝오버 확인, 공유 의도를 우선 측정한다.

### 9.2 핵심 이벤트 정의

| 이벤트 | 정의 |
|------|------|
| `install_completed` | 첫 실행 후 초기 상태 로딩이 완료된 시점 |
| `first_reaction_experienced` | 설치 후 24시간 내 충전 반응 또는 sleep/wake 반응을 1회 이상 본 시점 |
| `popover_opened` | 사용자가 메뉴바 아이콘을 눌러 팝오버를 연 시점 |
| `share_clicked` | 사용자가 공유 버튼을 누른 시점 |
| `share_copied` | 공유 문구가 실제로 클립보드에 복사된 시점 |

### 9.3 유지율 정의

| 지표 | 정의 |
|------|------|
| D1 Retention | 설치 다음 날 앱을 1회 이상 실행한 사용자 비율 |
| D7 Retention | 설치 후 7일차에 앱을 1회 이상 실행한 사용자 비율 |

### 9.4 초기 분석 질문

- 설치한 사용자가 24시간 내 첫 반응 경험까지 도달하는가
- 사용자는 팝오버를 반복적으로 열어 보는가
- 사용자는 자신의 상태나 성향을 공유하고 싶어 하는가
- 유지율은 단순 귀여움보다도 실제 반응 경험과 더 강하게 연결되는가

## 10. 릴리스 범위

### 10.1 Phase 0 — 이모지 프로토타입 (1주)

목표는 메뉴바 생존성 검증이다.

- 이모지(`😺`) 기반 메뉴바 아이콘
- 충전 / 수면 2개 센서만 연동
- 상태머신 없이 단순 반응

완료 기준:

- 충전선 연결 시 메뉴바 반응이 바뀐다
- 뚜껑 닫힘 / 열림에 따라 잠듦 / 기상 상태가 바뀐다

### 10.2 v1.0 — 살아있는 펫 검증

목표는 **"진짜 살아있는 느낌이 드는가"**를 검증하는 것이다.  
성향 고착이나 진화보다 먼저, 메뉴바에서 계속 켜둘 가치가 있는지를 본다.

포함:

- 안정적인 센서 4종: 전원 연결, 저배터리, sleep, wake
- 숫자 없는 내부 상태 리듬: 에너지 / 기분 / 피로
- 기본 상태 표현: idle / sleep / low-energy / wake
- 기본 팝오버
- 이벤트 로그

제외:

- 사용자 노출 숫자 HUD
- 성향 분기 공개
- 진화 연출
- CPU 기반 성향 판단
- 성향별 idle 변형
- 공유 버튼
- 슬랩 감지

완료 기준:

- 유저가 별도 조작 없이도 "얘가 살아있다"는 느낌을 받는다
- 메뉴바에서 거슬리지 않고 오래 켜둘 수 있다
- 팝오버가 숫자판이 아니라 상태 카드처럼 읽힌다
- 리소스 사용량 목표를 만족한다

### 10.3 v1.1 — 성향 감지 도입

목표는 MacTama의 차별점인 **"내 사용 습관이 펫 성향이 된다"**를 제품에 실어 올리는 것이다.

포함:

- 성향 분기 로직
- CPU 모니터링
- 시간대 기반 사용 패턴 분석
- 성향 배지 표시
- 공유 버튼

완료 기준:

- 유저가 자신의 펫을 하나의 유형으로 인식한다
- 성향 차이가 메뉴바와 팝오버에서 구분된다
- 공유 텍스트가 자연스럽게 생성된다

### 10.4 v1.2 — 진화 연출과 반응 차별화

포함:

- 진화 순간 연출
- 성향별 idle 변형
- 성향별 이벤트 반응 차별화

완료 기준:

- 진화가 단순 상태 변경이 아니라 감정 포인트로 느껴진다
- 메뉴바 14px에서도 변화가 식별된다

### 10.5 v2.0 — 확장

- Stage 2 완전 고착
- 주간 리포트
- 커스텀 이름
- 사운드팩 / 스킨
- 슬랩 카운터 및 고급 반응
- Apple Watch 연동 검토

## 12. 배포 전략

**초기 배포는 직접 배포(DMG 또는 ZIP) + Developer ID 서명 + notarization 조합을 기본으로 한다.**

- App Store 심사 대기 없이 빠르게 개선할 수 있다
- 메뉴바 유틸리티 특성상 직접 배포가 초기 실험에 유리하다
- 다만 macOS Gatekeeper 통과와 사용자 신뢰 확보를 위해 Developer ID 서명과 notarization을 전제로 한다
- 이 경로를 위해 Apple Developer Program 가입이 필요하며, 현재 기준 연회비는 `99 USD`다
- App Store 배포는 안정화 이후 보조 채널로 검토한다

## 14. v1 기술 수용 기준

### 14.1 센서 정확도

- 전원 연결/해제 이벤트는 실제 상태 변경 후 3초 이내 반영되어야 한다
- sleep/wake 이벤트는 누락 없이 상태 전환되어야 한다
- 저배터리 상태는 OS 배터리 상태와 사용자 체감상 어긋나지 않아야 한다

### 14.2 슬립/웨이크 복구

- 뚜껑 닫힘 시 sleep 상태로 전환되어야 한다
- 뚜껑 열림 후 5초 이내 정상 상태 또는 wake 애니메이션으로 복귀해야 한다
- sleep 중 타이머 누적이나 중복 반응이 없어야 한다

### 14.3 재시작 후 상태 복원

- 앱 재실행 시 마지막 저장 시점과 현재 시점의 경과 시간을 계산해 수치에 반영해야 한다
- 앱 강제 종료 후 재실행해도 상태가 초기화되지 않아야 한다
- 최근 이벤트 로그 3건은 복원되어야 한다

### 14.4 Idle 리소스 기준

- idle 상태 평균 CPU 사용량은 0.1% 미만이어야 한다
- 메모리 사용량은 20MB 미만이어야 한다
- 기본 폴링 기준에서 눈에 띄는 배터리 소모가 없어야 한다

## 16. 최종 요약

MacTama는 단순한 메뉴바 펫이 아니다.  
**맥북 사용 습관이 펫의 성향으로 굳어지는 앱**이다.

다만 첫 버전에서 증명해야 하는 것은 진화의 화려함보다,

> **메뉴바에 켜두면 정말 살아있는 것처럼 느껴지는가**

이다.

따라서 제품 비전은 성향 분기와 진화를 유지하되,  
실제 출시 범위는 v1에서 작게 시작하고 v1.1, v1.2로 확장한다.
